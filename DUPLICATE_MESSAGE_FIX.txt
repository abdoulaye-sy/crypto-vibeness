===========================================
DUPLICATE MESSAGE DISPLAY FIX
===========================================

PROBLEM:
--------
Messages were being displayed multiple times or with duplicated sender names.
Example: "asy: asy: cc imam" instead of "asy: cc imam"

ROOT CAUSE:
-----------
In client.py parse_and_display_message() function, there was problematic code at line 225:

    if sender and sender != "[SERVER]":
        # ... print formatted message
    else:
        print(text_content)  ← THIS LINE WAS THE PROBLEM!

This `else` block would execute when the sender check failed.
The problem: `text_content` contained the UNDECRYPTED message with both
the sender name AND the encrypted content. Printing it directly showed garbled output
and potential duplication of sender name.

Additionally, there was a fallback handler that would print anything that didn't have
a colon separator, which could cause unexpected output.

SOLUTION:
---------
Remove the fallback `else` block and the malformed message handler entirely. 
The logic now is:

1. If starts with [SERVER] → handle as system message → return
2. If starts with [DM from ...] → handle as received DM → return  
3. If starts with [DM → ...] → handle as sent DM → return
4. If has ':' and valid sender → handle as user message → return
5. Otherwise → silently ignore (no print, no fallback)

CODE CHANGES:
-------------
File: client.py
Lines: 190-221

-------

    if ':' in text_content:
        parts = text_content.split(':', 1)

        # ... message extraction and decryption ...

            # ... print formatted message ...
        else:

            print(text_content)  ← REMOVED: Printed undecrypted text

        # Just display as-is if can't parse
