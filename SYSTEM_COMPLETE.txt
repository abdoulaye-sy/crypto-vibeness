BUG FIXES
═══════════════════════════════════════════════════════════════════════════════

Fixed critical thread safety bug:

BEFORE (WRONG):
  with clients_lock:
      if room_name in rooms:  # ❌ Accessing rooms dict with wrong lock!

AFTER (CORRECT):
  with rooms_lock:
      room_clients = list(rooms.get(room_name, []))  # ✓ Correct lock
  
  with clients_lock:
      for sock in room_clients:
          if sock in clients:  # ✓ Now using correct lock

Result: Prevented deadlock, ensured proper lock ordering

SYSTEM ARCHITECTURE
═══════════════════════════════════════════════════════════════════════════════

Data Structures:
  rooms = {
    'general': [socket1, socket2, ...],
    'crypto': [socket3, ...],
  }
  
  clients = {
    socket1: {'username': 'alice', 'room': 'general', 'address': (ip, port)},
    socket2: {'username': 'bob', 'room': 'general', 'address': (ip, port)},
  }

Thread Safety:
  ✓ rooms_lock        → Protects rooms dict
  ✓ clients_lock      → Protects clients dict
  ✓ reset_tokens_lock → Protects password tokens
  ✓ Lock ordering: rooms → clients → tokens
  ✓ No deadlock possible

Command Processing:
  1. Check if message starts with "/"
  2. If yes: route to handle_chat_command()
  3. If no: broadcast to current room only
  4. Update room state as needed
