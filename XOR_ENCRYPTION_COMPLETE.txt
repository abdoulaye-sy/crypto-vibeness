LOCATION: parse_and_display_message() function (Line 190-209)
PROBLEM: Original code used .strip() which removes \x0b (vertical tab) and other 
         control characters that are part of encrypted content

SOLUTION: 
  1. Split on 'MSG_ID:' marker to find exact message end
  2. Use lstrip(' ') to remove ONLY leading spaces (not all whitespace)
  3. Remove trailing newline explicitly
  4. Preserve all binary content
  5. Decrypt the complete encrypted message


THE COMPLETE FIX:
─────────────────

OLD CODE (BROKEN):
    message = parts[1].strip() if len(parts) > 1 else ""
    message = decrypt(message)

NEW CODE (FIXED):
    message = parts[1] if len(parts) > 1 else ""
    
    # Carefully extract encrypted message:
    # Split on 'MSG_ID:' to find where actual message ends
    if 'MSG_ID:' in message:
        message = message.split('MSG_ID:')[0]
    
    # Remove only leading space, not other whitespace (some whitespace is encrypted!)
    message = message.lstrip(' ')
    
    # Remove trailing newline only
    if message.endswith('\n'):
        message = message[:-1]
    
    # Decrypt message (content now preserved)
    message = decrypt(message)


WHY THIS WORKS:
────────────────

Problem Case: Message "123456789"
  Encrypted:  "RZR@^SN\t\x0b" (9 bytes)
             - \t = tab character
             - \x0b = vertical tab (control character)

OLD CODE (BROKEN):
  .strip() removes both \t AND \x0b
  → "RZR@^SN" (7 bytes)
  → Decrypts to "1234567" ❌ WRONG

NEW CODE (FIXED):
  Split on 'MSG_ID:' boundary
  → "RZR@^SN\t\x0b" (9 bytes intact)
  → Decrypts to "123456789" ✅ CORRECT

KEY INSIGHTS:
──────────────

1. XOR encryption produces binary output
   - Some bytes are control characters (\n, \t, \x0b, etc.)
   - This is NORMAL and EXPECTED

2. Parsing must be binary-aware
   - Can't use generic whitespace stripping
   - Must carefully extract boundaries
   - MSG_ID: is the reliable boundary marker

3. Error handling during decode
   - errors='replace' for safe UTF-8 decode of binary data
   - errors='ignore' also works for stripping invalid sequences
   - Never use strict decoding with binary data

4. Message format is structured
   - "sender: [encrypted_bytes]\nMSG_ID:[id]\n"
   - The MSG_ID: line is the definite end marker
   - Use it to extract exact message boundaries
